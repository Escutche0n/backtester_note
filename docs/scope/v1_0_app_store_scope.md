# Backtester Note 1.0 App Store Scope

> 本文记录 Elvis 2026-04-26 口头裁定的 1.0 上架边界。  
> 它不是 PRD 替代品；若与 `docs/prd/Backtester_Note_PRD_v2.md` 冲突，以 PRD 为准，并把冲突升级给 Elvis 决定是否改 PRD。

---

## 1. 1.0 定位

1.0 是一个**本地优先、手动数据、可真实复盘国内公募基金持仓的 iOS app**。

1.0 不追求 Pro 自动化能力，不依赖自建后端，不用“自动拉取净值”作为上架前提。

---

## 2. 标的范围

- 只覆盖**国内公募基金**。
- 范围按“支付宝能覆盖的基金范围”理解。
- 1.0 不主动扩展到股票、海外基金、加密资产、期货、债券自定义品种等。
- ETF 是否按公募基金处理，等真实数据模型进入时再按基金代码和净值规则判断；不单独开股票式行情逻辑。

---

## 3. 数据输入

### 3.1 JSON 导入

1.0 必须支持：

- `json_import.v1` 文件导入
- 导入前预览
- schema / baseline / snapshot / flow 基础校验
- 用户确认后写入本地数据

### 3.2 手动净值

1.0 必须支持用户维护每日净值：

- 净值精度：4 位小数
- 粒度：按基金代码 + 日期
- 录入方式：至少支持单条手动录入；批量 JSON / CSV 可作为 1.0 加速项，但不能阻塞核心闭环
- 缺失净值时 UI 必须能解释“哪些指标暂不可算”，不能静默给假结果

---

## 4. 持仓复盘

1.0 必须支持真实持仓复盘，而不是只停留在 mock UI：

- 起始快照
- 交易流水
- 当前持仓
- 账户单位净值
- 持有收益
- XIRR
- 最大回撤
- 再平衡失衡比例

数据来源以本地 JSON / 手动录入为准。

---

## 5. 影子持仓

1.0 必须有影子持仓。

定义：

- 影子持仓是一个不影响真实账户的模拟组合 / 对照组合。
- 用于回答“如果我按另一套配置 / 纪律执行，历史上会怎样”。
- 影子持仓可与真实持仓做对比，但不能混入真实账户成本、份额和流水。

1.0 最小能力：

- 创建影子组合
- 配置基金与目标权重
- 基于同一批手动净值计算历史表现
- 与真实持仓在持仓页或回测页形成可读对比

---

## 6. 历史回测

1.0 历史回测必须完整覆盖四种模式：

1. 一直持有
2. 定投
3. 定期再平衡
4. 阈值再平衡

回测输入使用本地基金净值数据，不要求 Pro 自动拉取。

回测输出至少包含：

- CAGR
- XIRR
- 最大回撤
- 夏普
- 卡尔马
- 期末价值
- 与真实持仓 / 影子持仓的可比较结果

---

## 7. 持仓页 8 个状态 Graph

这里的“8 个 graph”是**App 内持仓页面的小状态组件**，不是 iOS 桌面 WidgetKit。

1.0 目标组件：

1. 单位净值
2. 当日 / 区间收益
3. 持有收益
4. XIRR
5. 最大回撤
6. 再平衡失衡比例
7. 风险 / 波动类指标
8. 影子持仓对比 / 超额表现

这些组件可以是小尺寸卡片、sparkline、bar 或 ring，不需要做成桌面 Widget。

---

## 8. 明确不进 1.0

以下功能不作为 1.0 上架前提：

- Pro 订阅
- 后端自动拉取基金净值
- 自动后台同步
- Pro 自动刷新 / 自动提醒
- 云备份
- 高级外观自定义
- 桌面 WidgetKit 的 8 个系统小组件
- 大规模多账户 / 多组合 Pro 限额能力

---

## 9. 开发节奏

Elvis 当前计划：

- 周一 / 周三 / 周五：视频主业
- 周二 / 周四 / 周六：App 开发
- 周日：轻 review / 排下周优先级，原则上不做重代码

工程执行规则：

- 每个开发日只做一个连贯小单元。
- 小单元结束必须 test / review / commit。
- 若已有未提交改动，先收口再开新功能。

---

## 10. 当前最高优先级

下一步仍然是 Phase 1c：

1. 最小 Persistence / PortfolioService
2. JSON import “确认写入”
3. 持仓页从 mock 数据切向本地真实数据的第一刀

原因：没有本地数据闭环，影子持仓、8 个状态 graph、历史回测都没有稳定数据来源。
