# Backtester Note PRD

## 1. 文档目标

本文档用于定义暂定 `Backtester Note` 的产品重构方向，目标是：

- 在不丢失现有产品核心算法和投资语义的前提下，重构一版更清晰、更稳定、更适合长期演进的 iOS 基金工具产品。
- 以前端体验升级和后端数据底座拆分为主线，重新梳理免费版与 Pro 版边界。
- 保留当前产品中最有价值的部分，尤其是持仓净值、趋势雷达、定投回测、历史回测和小组件能力。
- 让新产品在视觉风格上贴近 PeakWatch 的“高密度但克制、专业、信息优先”气质，同时整体交互与系统体验符合 iOS 26 的设计语言。

本文档优先服务以下工作：

- 产品方案统一
- UI 重设计
- 后端接口设计
- 商业化边界定义
- 后续技术拆解与排期

## 2. 产品背景

当前参考产品已经具备较完整的能力基础：

- 持仓主页与持仓交易流水
- 账户级净值曲线与收益分析
- 官方净值优先的 NAV 计算逻辑
- 趋势雷达 / 诊断评分
- 单基金 / 多基金回测
- 定投计划与历史回测记录
- 多个持仓相关小组件

当前问题不在于“功能不够多”，而在于产品结构仍偏工程堆叠，主要表现为：

- 前端功能入口较多，但主线价值没有被清晰聚焦。
- 数据获取目前仍偏客户端直连，后续在稳定性、缓存、任务调度、Pro 自动更新上存在明显天花板。
- 免费版与付费版的价值边界尚未产品化表达。
- UI 还没有形成足够鲜明、统一、可持续扩展的视觉语言。

因此这次不是简单加功能，而是一次“产品主线重构”：

- 主轴从“基金工具集合”收敛为“个人基金持仓操作系统”。
- 后端从“辅助请求层”升级为“统一数据与计算底座”。
- UI 从“功能页”升级为“持续观察、持续复盘、持续更新”的系统化体验。

## 3. 产品愿景

暂定 `Backtester Note` 的目标不是做一个泛基金行情 App，而是做一个面向个人投资者的、围绕真实持仓与策略复盘展开的基金投资工作台。

一句话定义：

> 帮用户持续记录持仓、理解账户净值变化、评估自己的投资动作，并通过回测和小组件形成长期复盘习惯。

产品应当让用户在三个问题上得到明确答案：

1. 我现在到底持有什么，账户真实状态如何？
2. 我的净值曲线和收益变化，究竟来自市场波动还是我自己的操作？
3. 如果继续按某种方式定投或调整组合，过去会发生什么，未来大概率又意味着什么？

## 4. 目标用户

### 4.1 核心用户

- 长期投资国内公募基金的个人投资者
- 已经有真实持仓，且愿意维护交易记录或持仓快照的人
- 对“账户净值”、“策略复盘”、“定投节奏”、“组合变化”敏感的人

### 4.2 次级用户

- 使用快捷指令、Excel、JSON 管理自己账户数据的进阶用户
- 想通过回测验证基金/组合/SIP 方案的人
- 希望用小组件每天轻量查看投资状态的人

### 4.3 非目标用户

- 以高频交易、实时行情、资讯消费为核心需求的人
- 需要券商交易接入、社交讨论、社区跟单的人
- 主要关注股票、期货、加密资产而非公募基金的人

## 5. 产品定位与原则

### 5.1 产品定位

暂定 `Backtester Note` 是一个：

- iOS 原生
- 以持仓为核心
- 以净值与复盘为主线
- 以本地体验为入口、以后端稳定数据为底座
- 面向长期投资场景的个人基金管理工具

### 5.2 核心原则

#### 原则 A：持仓优先

首页、主导航、Widget、后台更新都应围绕真实持仓展开，而不是围绕基金搜索或行情浏览展开。

#### 原则 B：算法口径延续

现有产品里已经建立的计算语义必须尽可能保留，尤其包括：

- NAV 净值曲线计算逻辑
- 交易流水对净值的影响处理
- 官方净值优先、盘中估值补充的口径
- 趋势雷达评分结构
- 定投回测与组合回测的周期、再平衡、基准比较逻辑

重构可以优化实现，不应轻易改变用户已经习惯的结果口径。

#### 原则 C：后端做底座，不做重平台

后端优先承接：

- 数据抓取
- 数据清洗
- 标准化 JSON 输出
- 缓存
- 后台任务
- 用户 Pro 自动刷新所需的同步能力

不优先做：

- 社交体系
- 复杂账户体系
- 多端复杂协作系统
- 大而全的管理后台

#### 原则 D：免费可用，Pro 明确增益

免费版必须能完整体验产品方法论，但在自动化与省心程度上保留明显升级空间给 Pro。

#### 原则 E：系统级体验

目标不是做“几个页面”，而是形成包括：

- App 主体验
- 小组件
- 快捷指令输入
- 后台自动更新
- 本地通知

在内的一整套 iOS 生态体验。

## 6. 版本策略

### 6.1 免费版

免费版支持用户通过手动方式建立完整使用闭环。

包含：

- 手动搜索基金与查看历史数据
- 手动录入持仓快照与交易流水
- 通过快捷指令 + JSON 导入数据
- 查看账户净值、收益、趋势雷达、历史回测
- 使用基础小组件
- 本地保存回测历史与持仓分析结果

免费版定位：

- 让用户确认产品价值
- 让重度用户通过自己的数据源接入
- 保持可玩性与可验证性

### 6.2 Pro 版

Pro 版的核心卖点不是“多几个页面”，而是“持续自动化”。

包含：

- 自动后台更新基金数据
- 自动刷新持仓估值 / 官方净值
- 自动生成小组件所需快照
- 更高频率的后台刷新策略
- 更完整的历史缓存与更稳定的数据同步
- 后续可扩展的高级诊断与高级回测参数

Pro 版定位：

- 帮用户省掉手动更新与快捷指令编排成本
- 让持仓主页和 widgets 真正变成“每天可直接看”的系统

## 7. 产品结构建议

建议新版本信息架构从当前多模块并列，收敛为四个主层级：

### 7.1 今日

这是新的默认首页，强调“今天该看什么”。

包含：

- 账户总览卡片
- 今日估算收益 / 最新官方净值变化
- 组合净值曲线摘要
- 趋势雷达摘要
- 最近异常提醒
- 最近一次回测 / SIP 计划摘要
- 小组件同步状态

目标是让用户打开 App 后 10 秒内知道：

- 今日账户状态
- 当前是否偏离预期
- 是否需要进一步点进查看

### 7.2 持仓

这是产品真正的核心工作区。

包含：

- 持仓总览
- 单基金持仓详情
- 交易流水
- 账户净值
- 相对基准表现
- 成本、持有收益、回撤、仓位结构
- 持仓诊断与对账状态

### 7.3 回测

统一承接：

- 单基金历史回测
- 组合历史回测
- 定投回测
- 再平衡回测
- 已保存历史回测

回测模块的价值不是展示复杂图表，而是让用户可以复用“自己真实持仓里正在考虑的问题”。

### 7.4 组件

集中管理：

- 已启用 widgets
- 小组件数据状态
- 最近快照更新时间
- 快捷指令 / JSON 导入模板
- Pro 自动同步说明

这个页面本质上是“系统集成层”，让用户知道产品如何和 iOS 生态协同工作。

## 8. 核心功能定义

## 8.1 持仓与 NAV

### 目标

让用户看到的是“账户真实净值”，不是简单的收益数字拼接。

### 必须保留的能力

- 账户级 NAV 曲线
- 基于交易流水的净值折算
- 外部现金流与收益变化分离
- 官方净值优先，盘中估值作为补充
- 当流水不足时，用快照模式降级展示并清楚标记可信度

### 建议定义

- `确认净值`：基于官方历史净值计算的账户净值点
- `盘中估值`：仅用于当日预估变化，不覆盖历史确认值
- `账户净值`：反映账户在剔除外部申赎现金流后自身表现的统一指标
- `可信度状态`：已对账 / 待核对 / 仅快照 / 仅流水

### 用户价值

- 用户终于能回答“我到底赚了多少”以及“这是不是因为我自己加仓导致的”
- 相较泛基金工具，形成强差异化

## 8.2 趋势雷达 / 投资诊断

### 目标

把现有趋势雷达从“一个有趣功能”提升为“持仓复盘核心框架”。

### 必须保留的能力

- 六维雷达结构
- 当前分数与历史对比
- 每日总分趋势
- 对用户投资动作的定性解释

### 建议重构方向

- 把雷达从单纯“评分可视化”升级为“投资行为复盘层”
- 每一维都给出更明确的解释性文案
- 强化“较上周 / 较上月 / 较最近一次调仓”的对比语义
- 支持在详情页展开看到各维度分数变化的原因说明

### 产品表达建议

雷达不应被表达成“预测工具”，而应表达成：

- 当前操作质量
- 组合管理纪律
- 投资节奏稳定性
- 风险控制状态

## 8.3 定投回测

### 目标

保留当前定投回测能力，并让它和真实投资行为更紧密关联。

### 必须保留的能力

- 多种定投周期
- 周几 / 月几号等规则
- 首笔金额 + 每期金额
- 组合定投
- 再平衡开关
- 再平衡阈值
- 基准比较
- CAGR / XIRR / 回撤等指标

### 建议增强

- 支持从真实持仓一键带入参数
- 支持从当前组合直接发起回测
- 支持“如果从某次历史买入日起继续定投”的场景
- 结果页突出“最大回撤区间”和“定投执行记录摘要”

## 8.4 历史回测

### 目标

让回测记录成为“复盘资产”，而不是一次性结果。

### 必须保留的能力

- 单基金回测
- 组合回测
- 基准对比
- 保存回测历史
- 再打开查看历史参数与图表

### 建议增强

- 增加“最近回测”和“收藏回测”
- 允许对两个历史回测结果做快速对比
- 回测记录中保留当时参数快照，保证结果可复现

## 8.5 数据导入

### 免费版主方案

通过快捷指令 + JSON 导入，是现阶段最合适的免费版方案。

原因：

- 不引入复杂账户系统
- 适合高级用户自己拼接数据
- 与 iOS 生态天然兼容
- 可以较低成本承接不同来源的数据

### 导入对象

- 持仓快照
- 交易流水
- 基础账户元信息

### 导入原则

- JSON schema 必须稳定、可公开、可版本化
- App 内要提供模板、示例和错误提示
- 导入后必须有对账与预览步骤，避免污染持仓

## 8.6 Widgets

### 必须保留的现有 widgets

- 组合净值曲线 Widget
- 趋势雷达 Widget
- 每日总分 Widget

### 建议扩展的 widgets

- 今日收益摘要 Widget
- 单账户持仓状态 Widget
- 定投计划提醒 Widget
- 最新异常提醒 Widget

### Widget 设计原则

- 不做资讯卡片式堆砌
- 每个组件只回答一个问题
- 与首页和持仓页的视觉语言保持一致
- 在 Pro 版下保证更稳定的后台刷新

## 9. 视觉与交互方向

## 9.1 风格关键词

目标风格建议使用以下关键词：

- PeakWatch 式的信息密度与仪表感
- iOS 26 的系统材质与层级感
- 专业、克制、冷静
- 卡片更少，分区更清晰
- 图表和数字优先于装饰元素

## 9.2 视觉表达建议

- 首页采用“信息仪表盘”式结构，而不是传统 feed。
- 强化大数字、净值曲线、状态标签的优先级。
- 使用更明确的层级分区，让“今日状态 / 持仓 / 诊断 / 回测入口”自然下沉。
- 重要图表采用更轻的背景和更强的走势识别，而不是大面积拟物。

## 9.3 iOS 26 对齐建议

设计上应尽可能贴合当前系统语言：

- 使用更系统化的材质层级和玻璃感表面
- 强化圆角连续性、浮层感和滚动中的层次变化
- 更自然地整合大标题、分段控件、图表和底部工具区
- 保持原生动态与 Widget 之间的语义一致

注意：

- 这里应是“符合 iOS 26 的原生气质”，而不是生硬堆叠视觉效果
- 投资工具最重要的仍然是可读性、稳定性和信息密度

## 10. 后端方案建议

## 10.1 后端定位

新后端应继续保持轻量，但能力边界要比当前更完整。

建议定位为：

> 面向 `Backtester Note` App、快捷指令与后台同步任务的统一基金数据与组合计算服务

## 10.2 后端职责

### 应承担

- 基金搜索
- 实时估值获取
- 历史净值抓取
- 数据清洗与字段标准化
- 组合历史净值聚合
- 组合实时估值聚合
- 缓存与失败回退
- Pro 用户后台更新任务
- Widget 快照所需聚合接口

### 暂不承担

- 复杂用户系统
- 完整 SaaS 后台
- 社交分享平台
- 重型策略引擎平台化

## 10.3 建议 API 分层

### 第一层：基金基础数据

- `fund/search`
- `fund/realtime`
- `fund/history`
- `fund/availability`

### 第二层：组合与持仓计算

- `portfolio/realtime`
- `portfolio/history`
- `portfolio/nav`
- `portfolio/radar`

### 第三层：导入与同步

- `import/validate`
- `import/preview`
- `sync/run`
- `widget/snapshot`

## 10.4 数据与计算原则

- 客户端展示口径尽量统一依赖后端标准字段
- 但核心账户语义应与现有客户端算法保持一致
- 重要指标必须可复现，不允许同一账户在不同端口径漂移

推荐做法：

- 先把当前 iOS 里的关键计算口径文档化
- 再决定哪些上收到后端，哪些仍保留在端上

建议短期方案：

- 历史净值抓取、搜索、实时估值统一后端化
- 持仓 NAV、趋势雷达仍先沿用端上逻辑
- 待口径冻结后，再逐步将组合级聚合迁移到后端

这样最稳妥，因为它最大程度降低“重构导致结果变化”的风险。

## 11. 商业化策略

## 11.1 免费版价值

免费版必须足够完整，能让用户真正形成习惯。

建议免费版包含：

- 手动维护持仓
- 快捷指令 + JSON 导入
- 核心净值曲线
- 核心趋势雷达
- 基础回测
- 基础 widgets

## 11.2 Pro 版价值

Pro 版要卖“省心 + 持续更新 + 系统存在感”。

建议 Pro 版首批权益：

- 自动后台拉取净值 / 估值
- 自动更新 Widget 数据
- 自动生成每日持仓摘要
- 更长历史缓存
- 更稳定同步与失败回退

后续可追加权益：

- 高级雷达解释
- 高级回测参数
- 更丰富的组合对比
- 更高频提醒与自动化

## 12. 成功指标

第一阶段建议关注以下指标：

- 新用户完成首次导入或首次持仓录入的比例
- 7 日内再次打开持仓首页的比例
- 使用至少一个 widget 的用户占比
- 完成一次回测的用户占比
- 免费用户升级 Pro 的转化率
- Pro 用户后台同步成功率
- Widget 快照更新成功率

## 13. 分阶段路线图建议

## Phase 1：产品骨架重构

- 完成新的信息架构
- 重做首页与持仓主页
- 固化 JSON 导入协议
- 接入轻量后端的基金搜索 / 实时 / 历史 API
- 保留现有 NAV 和回测核心口径

## Phase 2：自动化能力上线

- 上线 Pro 后台自动更新
- 补齐 widget 快照接口
- 优化缓存和失败回退
- 增加同步状态与错误提示

## Phase 3：复盘能力增强

- 升级趋势雷达解释
- 加强历史回测管理
- 增加回测结果对比
- 打通持仓与回测之间的参数联动

## 14. 最终建议

如果以“怎么做更合适”为标准，最合适的方案不是推倒重来，而是：

1. 保留当前产品最有价值的算法和使用心智，尤其是持仓 NAV、趋势雷达、定投回测、历史回测、小组件。
2. 用一个轻量但更稳定的后端，接管数据抓取、标准化、缓存和 Pro 自动更新。
3. 用一次偏系统级的 iOS 26 风格重设计，把首页、持仓、回测、组件四层结构重新组织。
4. 免费版坚持“手动可完整使用”，Pro 版明确售卖“自动化与持续更新”。

这条路线的优点是：

- 风险低
- 保留已有产品资产
- 商业化边界清晰
- 技术迁移路径可控
- 最终体验会比现在更像一个成熟产品，而不是功能集合
